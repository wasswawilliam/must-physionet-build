Guidelines for reviewers

When reviewing content, check that the requirements below are met. If not, you can copy the relevant points below into your message to the authors, preceded by "Please ensure that:".

All projects

  • A README file is included alongside the files. At minimum, this file should include a title and a brief description of the package content.
  • Sensitive data and protected health information have been removed.
  • Files should be clearly named and must not include spaces (use underscores instead for increased readability) or special characters (e.g. "/","\,"."). Further, filenames should generally be lowercase (exceptions are "special files‚Äù, such as the RECORDS and ANNOTATORS files used for waveforms). Please use brevity when naming files (e.g. 1001_abp.dat is preferable to subject_1001_ABP_wave_100Hz.dat).
  • Files should be in open-source format and machine readable. Proprietary formats such as MatLab (*.mat), Excel (*.xls) or Microsoft Word (*.doc) should be converted to an open-source format.

Data (general)

  • Comma-Separated Value (CSV) formatted files are a good option for small datasets. CSVs should be formatted according to the RFC 4180 specification.
  • Tabulated data is arranged such that each variable is a column and each observation (or case) is a row ("tidy data").

Data (waveforms)

  • In general, high time-resolution data such as ECG and EEG recordings should be stored in a WFDB compatible (or other open-source) format where possible. Details such as gain and baseline should be included in the file headers.
  • The submitted project should include sufficient information to visualize waveforms in LightWave (i.e. RECORDS, *.hea, etc. files). The *.hea files should contain descriptive signal names (i.e. no repeated signal names inside of the the same *.hea file).

Software

  • Instructions for installation and usage should be clearly documented.
  • Dependencies should be indicated in a requirements file or similar.
  • Unit tests should be used to demonstrate correct functioning of major features of the software.
  • Standard style guidelines should be followed where appropriate (for example, PEP8 for Python).

Challenges

  • The objective of the Challenge is clearly stated.
  • A timeline is provided, including a start date, final submission date, and scoring data.
  • The criteria for assessment are clear and unambiguous.
